# GitHub Issue Rules

## Mandatory Requirements

### Before Creating an Issue

1. **Search for duplicates** - Check if issue already exists
2. **Verify it's actionable** - Clear problem or request
3. **Provide context** - Enough information for others to understand

### Issue Content Requirements

1. **Clear Title**
   - Descriptive and specific
   - Summarizes the issue in one line
   - No vague titles like "it doesn't work"

2. **Complete Description**

   **For Bug Reports:**
   - Clear problem statement
   - Steps to reproduce
   - Expected vs actual behavior
   - Environment details (OS, versions, etc.)
   - Error messages or logs
   - Screenshots if relevant

   **For Feature Requests:**
   - Problem statement (what problem does this solve?)
   - Proposed solution
   - Alternatives considered
   - Use cases

   **For Documentation:**
   - What's missing or unclear
   - Suggested improvements
   - Location in docs

3. **Proper Categorization**
   - Add appropriate labels (bug, enhancement, documentation, question)
   - Assign to milestone if applicable
   - Set priority if relevant

### Issue Management

1. **Assignment**
   - Assign to yourself when starting work
   - Unassign if you can't complete it

2. **Updates**
   - Comment when starting work
   - Update on significant progress or blockers
   - Link related PRs

3. **Closing**
   - Close via PR with "Closes #123" or "Fixes #123"
   - Or close manually with explanation if not resolved via PR

## Anti-Patterns to Avoid

❌ Vague or one-line descriptions
❌ Multiple unrelated issues in one
❌ Missing reproduction steps for bugs
❌ No environment information
❌ Feature requests without explaining the problem
❌ Duplicate issues without referencing original
❌ Closing issues without resolution

## Best Practices

✅ One issue per problem or request
✅ Include minimal reproducible example for bugs
✅ Add screenshots or GIFs for UI issues
✅ Reference related issues or PRs
✅ Use templates if repository provides them
✅ Be respectful and professional
✅ Provide error messages and stack traces
✅ Update issue if you find more information

## Issue Lifecycle

1. **Open** - Issue created
2. **Triaged** - Labels and priority assigned
3. **Assigned** - Someone is working on it
4. **In Progress** - Work has started
5. **In Review** - PR is under review
6. **Closed** - Resolved or completed
